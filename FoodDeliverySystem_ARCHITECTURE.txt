Online Food Ordering System - Architecture Plan
==================================================

Project Name
------------
FoodDeliverySystem

Purpose
-------
FoodDeliverySystem is a C++ backend application for an online food ordering platform.
The system allows users to browse stores/restaurants, view food items and prices,
place orders, track preparation and delivery time, and communicate through live chat.

The long-term goal is to build a modular backend in C++ with a database, real-time
communication, asynchronous networking, and a web-based user interface.


1. High-Level Architecture
--------------------------

The application will be divided into the following main layers:

1. Frontend UI Layer
   - HTML
   - CSS
   - JavaScript
   - AJAX / Fetch API
   - WebSocket client for real-time updates and live chat

2. Backend Network Layer
   - C++
   - uWebSockets for asynchronous HTTP and WebSocket communication
   - REST-style endpoints for normal requests
   - WebSocket endpoints for real-time features

3. Application Service Layer
   - User management
   - Store management
   - Menu management
   - Order management
   - Delivery time estimation
   - Live chat handling
   - Notification handling

4. Data Access Layer
   - PostgreSQL database
   - C++ database access wrapper
   - Prepared statements
   - Transaction handling

5. Database Layer
   - PostgreSQL tables for users, stores, menu items, orders, order items,
     chat messages, addresses, and delivery estimation data

---------------------

Backend Language:

Networking:

  - HTTP endpoints

- Boost.JSON can be used for JSON parsing/serialization.
- Boost.DateTime or std::chrono can be used for time calculations.

- WebSocket API

Build System:
- Visual Studio project initially
- Later optional migration to CMake for cleaner dependency management


3. Important Design Decision
----------------------------

The project originally started with MySQL. Moving to PostgreSQL is acceptable and valuable,
but it means the database connection layer must be rewritten.

Recommended approach:
- Keep the application architecture database-independent.
- Create repository classes such as UserRepository, StoreRepository, and OrderRepository.
- Do not write SQL directly inside UI or business logic classes.
- This makes it easier to change database systems later if needed.


4. Main Features
----------------

4.1 User Features
-----------------
- Register account
- Login
- View available stores/restaurants
- View food menu and prices
- Add food items to cart
- Place an order
- View order status
- Receive estimated preparation and delivery time
- Use live chat for order support

4.2 Store/Admin Features
------------------------
- Add/edit/remove food items
- Update prices
- View incoming orders
- Accept or reject orders
- Update order preparation status
- Send messages to customers

4.3 Delivery Features
---------------------
- Store customer address
- Estimate delivery time
- Track order status
- Calculate total estimated time:
  preparation time + delivery time

4.4 Real-Time Features
----------------------
- Live chat between user and support/store
- Real-time order status updates
- Optional real-time notification when order is accepted, prepared, or dispatched

8. Order Status Flow
--------------------

Possible order statuses:
- pending
- accepted
- preparing
- ready_for_pickup
- out_for_delivery
- delivered
- cancelled

Typical flow:
pending -> accepted -> preparing -> ready_for_pickup -> out_for_delivery -> delivered

Real-time WebSocket updates should be sent whenever an order status changes.


9. Delivery Time Estimation
---------------------------

The estimated total time can start simple and become more advanced later.

Initial formula:
estimated_total_time =
    max(preparation time of all items) + delivery travel time + extra buffer

Example:
- Margherita Pizza preparation: 15 minutes
- Burger preparation: 12 minutes
- Delivery travel time: 20 minutes
- Buffer: 5 minutes

Estimated total:
max(15, 12) + 20 + 5 = 40 minutes

Possible future improvements:
- Use store workload
- Use number of active orders
- Use distance from store to user
- Use traffic data from an external API
- Use average historical preparation times


10. API Design
--------------

10.1 Auth API
-------------
POST /api/register
Request:
{
  "username": "john",
  "email": "john@example.com",
  "password": "secret"
}

POST /api/login
Request:
{
  "email": "john@example.com",
  "password": "secret"
}

10.2 Store API
--------------
GET /api/stores
Returns all available stores.

GET /api/stores/{storeId}
Returns details for one store.

10.3 Menu API
-------------
GET /api/stores/{storeId}/menu
Returns menu items for a store.

10.4 Order API
--------------
POST /api/orders
Creates a new order.

GET /api/orders/{orderId}
Returns order details.

PATCH /api/orders/{orderId}/status
Updates order status.

10.5 Chat API
-------------
GET /api/orders/{orderId}/messages
Returns previous chat messages for an order.


11. WebSocket Design
--------------------

WebSocket endpoint:
ws://localhost:9001/ws

Message types:
- chat_message
- order_status_update
- user_joined_chat
- user_left_chat
- notification

Example chat message:
{
  "type": "chat_message",
  "orderId": 15,
  "senderId": 3,
  "message": "Hello, is my order almost ready?"
}

Example order status update:
{
  "type": "order_status_update",
  "orderId": 15,
  "status": "preparing",
  "estimatedTotalMinutes": 35
}


12. Security Requirements
-------------------------

Minimum security requirements:
- Store password hashes, not plain passwords
- Use prepared statements
- Validate all user input
- Use HTTPS in production
- Use secure authentication tokens
- Do not expose database credentials in source code
- Use environment variables or configuration files ignored by Git
- Add rate limiting later
- Add role checks for admin/store actions

15. Final Notes
---------------

The project idea is valuable, but it is ambitious. The best approach is to avoid building
everything at once.

Recommended immediate next step:
- Do not start with uWebSockets, WebSockets, Boost, and frontend all at the same time.
- Start with the database schema, models, repositories, and console-based services.
- After the core business logic works, add HTTP endpoints.
- After HTTP endpoints work, add WebSockets.
- After the backend works, improve the frontend.

This approach avoids confusion and makes the system easier to test.
